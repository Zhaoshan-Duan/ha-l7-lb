What this implies for LB design

  • ›Every service boundary is an LB hop. Most are parallel fan-outs, not a chain — but the critical path from tap to confirmation still crosses 20–30 routing decisions.
  • ›Fan-out punishes the tails. Wait for 20 parallel calls and you get the slowest one. A single lazy backend anywhere in the tree stalls the whole order. Routing quality shows up in p99, not p50.
  • ›Health must converge faster than failures propagate. A stale UP flag on one LB leaks failures down the chain before peers notice.
  • ›HA-L7-LB is a rigorous reference for one such hop — measured, not guessed.

// 02What we are building

A custom L7 load balancer that survives instance failure, scales horizontally, and shares health state across replicas — without depending on a managed offering like ALB.

Goal · 01

High availability

Tolerate backend failures via active health checks + idempotent retry on a fresh, healthy target.

Goal · 02

Distributed state

Replicas converge on backend health via Redis Pub/Sub — no leader, no consensus, eventual sync.

Goal · 03

Pluggable policy

Round-robin, least-connections (P2C), and weighted routing behind one algorithms.Rule interface.

// 07Algorithm · Round Robin

The fair, boring baseline every other algorithm has to beat.

// lock-free · O(1) per call

next := atomic.AddUint64(&r.next, 1)
idx  := (next - 1) % uint64(len(servers))

return servers[idx].ServerURL
how

A single atomic.AddUint64 on a shared counter, modulo the number of healthy backends. No locks, no per-server state, no tracking.

why it works

A lock-free atomic add costs roughly 1 ns on modern x86 — effectively free. Distribution is mathematically exact: over any N requests, each healthy backend gets ⌊N/M⌋ or ⌈N/M⌉ hits. No randomness to worry about.

trade-offs
  • zero cross-LB coordination — scales trivially
  • deterministic — easy to reason about and test
  • ignores real server load — a slow backend still gets equal share
  • poor fit for heterogeneous fleets or long-lived connections

cost~1 ns per call
syncnone
best foruniform fleets

// 08Algorithm · Least Connections (P2C)

Two dice rolls beat a full scan — by an exponential margin.

// Power of Two Choices (Mitzenmacher, 2001)

i := rand.Intn(len(servers))
j := rand.Intn(len(servers) - 1)
if j >= i { j++ }  // ensure j != i

if servers[i].GetActiveConnections() <=
   servers[j].GetActiveConnections() {
    return servers[i].ServerURL
}
return servers[j].ServerURL
how

Pick two distinct random backends. Compare their atomic.Int64 active-connection counters. Route to whichever has fewer. Three ops on the hot path.

why it works

Azar · Broder · Karlin · Upfal (1994): with just 2 random picks and a min rule, the expected max load is bounded by log log N + O(1) — exponentially better than a single random pick. And it holds with purely local counters, no cross-LB sync.

trade-offs
  • near-optimal fairness with zero coordination cost
  • absorbs heterogeneous workloads automatically
  • slightly worse than RR on perfectly uniform loads (randomness variance)
  • requires maintaining per-server connection counters

cost2 rand + 2 loads
synclocal only
best forlong conns · load skew

// 09Algorithm · Weighted

Ratios that actually hold over time — without bursty hot-spots.

// random within epoch · decrement on hit

for len(candidates) > 0 {
    ri := rand.Intn(len(candidates))
    c  := candidates[ri]
    if wrr.Weights[c][1] > 0 {
        wrr.Weights[c][1]--
        return c
    }
    candidates = remove(candidates, ri)
}
resetEpoch()  // all exhausted
how

Each backend holds [orig, remaining]. Pick a random candidate; if it still has weight, decrement and return. Otherwise drop it and retry. When every counter hits zero, reset and start a new epoch.

why it works

Random picks within each epoch kill the bursty hot-spot pattern a strict sequential walker would produce. Over one epoch, weights 70/20/10 translate to exactly 70/20/10 requests — no approximation.

trade-offs
  • ratios hold exactly — no drift over time
  • natural fit for heterogeneous tiers (strong vs weak)
  • sync.Mutex serializes calls under very high QPS
  • static ratios only — doesn't react to real-time load

costO(N) worst
syncsync.Mutex

// 13Distributed state

Redis Pub/Sub sync

Health transitions publish to a shared channel. Peers subscribe and update their local InMemory pool. Debounced to avoid redundant writes.

  • ›Redis is optional — degraded mode runs local-only.
  • ›Periodic re-sync ticker reconciles drift.
  • ›Fire-and-forget publish keeps the request path fast.

Cloud Map DNS discovery

One DNS watcher per configured backend endpoint, polling every 5s. SyncServersBySource reconciles only that source — strong and weak tiers coexist.

  • ›No static config — backends come and go dynamically.
  • ›Per-source weight enables heterogeneous fleets.
  • ›Removed backends drain instead of vanishing.

// 15Backend endpoints

Five endpoints on every backend task. /api/compute was added as a deliberate design decision — a CPU-bound workload that produces measurable latency differences between strong and weak tiers and stresses the LB under real compute cost.

GET · liveness

/health

Unconditional 200 OK. Polled by the LB health checker; any non-200 or timeout marks the backend DOWN.

GET · POST · baseline

/api/data

Primary workload. 5–25 ms jitter latency. Accepts chaos headers X-Chaos-Error and X-Chaos-Delay.

GET · KEY DECISION

/api/compute

Iterated SHA-256 hashing. Default 50k iter, max 500k. Burns CPU for ~100–300 ms per request on Fargate.

GET · bandwidth

/api/payload

~1 MB JSON body. Exercises the proxy's streaming path (io.Copy) and verifies large transfers replay safely.

GET · long-lived

/api/stream

Chunked transfer · 10 chunks · 200 ms apart. Holds the proxy connection open to test LC accuracy under concurrent load.
